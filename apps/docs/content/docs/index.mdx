With React Server Components, your component renders once on the server. There's no re-render from "loading" to "loaded," so patterns like `{data || <Skeleton />}` can't handle loading states. Instead, you write a separate skeleton component for every piece of UI and pass it as a Suspense fallback.

Bones skips the duplication. You write your markup once and it handles both states. The skeleton and the real UI are the same component, so they can't drift apart.

## How it works

`createBones` accepts data or a promise of data. While loading, its `bone` function returns HTML attributes that style elements as skeletons via CSS. Once the data resolves, `bone` returns an empty object and your component renders normally. There are no hooks and no context providers.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4 comparison-grid">
<div>

```tsx title="PokemonCard.tsx"
import { createBones } from "@lovo/bones";

function PokemonCard() {
  const { bone, data, repeat } = createBones(fetchPokemon());

  return (
    <div className="card">
      <img
        className="card-image"
        src={data?.sprite}
        alt={data?.name ?? "Pokemon"}
        width={120}
        height={120}
        {...bone("block")}
      />
      <h3 className="card-name" {...bone("text", { length: 6 })}>
        {data?.name}
      </h3>
      <div className="card-types">
        {repeat(data?.types, 2, (type, i) => (
          <span key={type ?? i} className="badge" {...bone("text", { contained: true, length: 7 })}>
            {type}
          </span>
        ))}
      </div>
    </div>
  );
}
```

## Key ideas

- Works in Server Components. No hooks, no context, no `'use client'`.
- One component handles both loading and loaded states.
- Pass data or a promise. Bones wires up Suspense for you.
- Skeletons are pure CSS, themed with custom properties.
- Loading elements get `aria-busy="true"` automatically.

## Why not just add a `loading` prop?

You could keep both states in a single component by adding a `loading` prop and branching on it:

```tsx title="Single component with loading prop"
function PokemonCard({ loading }: { loading?: boolean }) {
  const data = loading ? undefined : use(fetchPokemon());

  return (
    <div className="card">
      {loading ? <div className="skeleton skeleton-image" /> : <img src={data.sprite} />}
      {loading ? <div className="skeleton skeleton-title" /> : <h3>{data.name}</h3>}
      {loading
        ? Array.from({ length: 2 }, (_, i) => <div key={i} className="skeleton skeleton-badge" />)
        : data.types.map((t) => <span key={t}>{t}</span>)}
    </div>
  );
}
```

This avoids a separate file, but every element still has two branches that can drift apart. `loading` also has to be passed through every layer of the tree. If a user card lives inside a sidebar inside a dashboard, all three components need the prop even though only the card uses it.

With Bones, none of that is needed. A component at any depth just calls `createBones()` and gets skeletons:

```tsx title="Loading prop — two branches per element"
function PokemonCard({ loading }: { loading?: boolean }) {
  const data = loading ? undefined : use(fetchPokemon());

  return (
    <div className="card">
      {loading ? <div className="skeleton skeleton-image" /> : <img src={data.sprite} />}
      {loading ? <div className="skeleton skeleton-title" /> : <h3>{data.name}</h3>}
      {loading
        ? [1, 2].map((i) => <div key={i} className="skeleton skeleton-badge" />)
        : data.types.map((t) => <span key={t}>{t}</span>)}
    </div>
  );
}

// Page.tsx — you still manage loading state
function Page() {
  return (
    <Suspense fallback={<PokemonCard loading />}>
      <PokemonCard />
    </Suspense>
  );
}
```

## Multiple async components

You can wrap multiple components in a single `<Suspense>` boundary so they resolve together. But you still need a skeleton component for each one in the fallback.

<div className="grid grid-cols-1 md:grid-cols-2 gap-4 comparison-grid">
<div>

```tsx title="Suspense — one boundary, three skeletons"
function Dashboard() {
  return (
    <Suspense
      fallback={
        <>
          <UserCardSkeleton />
          <StatsSkeleton />
          <ActivityFeedSkeleton />
        </>
      }
    >
      <UserCard />
      <Stats />
      <ActivityFeed />
    </Suspense>
  );
}
```

</div>
<div>

```tsx title="Bones — no skeletons to maintain"
function Dashboard() {
  return (
    <Bones>
      <UserCard />
      <Stats />
      <ActivityFeed />
    </Bones>
  );
}
```

</div>
</div>

`<Bones>` renders the same component tree as its own Suspense fallback, with a loading flag set via `React.cache()`. Every `createBones()` call in the tree sees that flag and shows skeletons. The children don't even know they're in a fallback. React Context could do the same thing, but a context provider is a client component. Adding `'use client'` defeats the point.

All components inside a `<Bones>` boundary show skeletons until the slowest promise resolves. If you want components to load independently, wrap each one in its own `<Bones>`:

```tsx
function Dashboard() {
  return (
    <>
      <Bones>
        <UserCard />
      </Bones>
      <Bones>
        <Stats />
      </Bones>
      <Bones>
        <ActivityFeed />
      </Bones>
    </>
  );
}
```
